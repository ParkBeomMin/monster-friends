# 방치형 오토배틀러 게임 기획서 (가제: 몬스터 프렌즈)

- 작성일: 2026-07-03
- 상태: 초안 (브레인스토밍 확정본)

---

## 1. 배경과 목표

### 1.1 왜 이걸 만드는가
기존 프로젝트 `덕뮤`(덕질 커뮤니티)는 기능은 충분했으나 **콜드스타트 문제**로 실패했다.
유저가 직접 콘텐츠(글)를 생성해야 가치가 생기는 구조 → 텅 빈 커뮤니티 → 죽음의 악순환.

이번 프로젝트는 그 실패를 정면으로 회피한다:
- **혼자서도 첫날부터 재밌다** (단일 플레이 PvE가 메인) → 콜드스타트 없음
- 만드는 사람 = 실제 유저(제작자가 오토배틀러 유저) → "뭘 만들어야 하는지" 안다

### 1.2 목표 (우선순위 순)
1. **만들고 싶은 게임을 만든다** — 제작자가 즐기는 오토배틀러 + 방치형. (이게 1순위, 창작 동기가 진짜 목표)
2. **용돈 수준의 부수입** — 현실적 목표치 월 20만원 안팎(광고 기준). 대박이 아니라 지속 가능한 소득.

### 1.3 제약 조건
- **1인 개발**, 주 5시간 안팎 (부업)
- 신규 언어/엔진 학습 최소화 (기존 웹 스택 재사용)
- 아트는 직접 그리지 않는다 → **에셋 팩 구매·조합**

## 2. 핵심 컨셉

> **"귀여운 픽셀 몬스터를 모아 잃어버린 지역을 되찾는 방치형 오토배틀러"**

미티파티(오토배틀러 조합의 재미) + 메이플 감성(귀여운 픽셀 지역·몬스터) + 방치형(오프라인 성장)의 교차점.
PvP는 **실시간이 아닌 비동기**로 구현해 콜드스타트와 서버 부담을 제거한다.

### 2.1 핵심 루프
```
스쿼드 편성 → 자동 전투로 스테이지(지역) 진행(PvE)
   → 벽에 막힘 → 방치로 자원 파밍 → 영웅/지휘관 강화 → 돌파
   → (곁다리) 아레나에서 다른 유저 스쿼드에 비동기 도전
```

## 3. 게임 시스템

### 3.1 유닛(몬스터) 설계 — 3축
1. **역할(Role)**: 🛡️ 탱커 / ⚔️ 근접딜 / 🏹 원거리딜 / ✨ 힐러·서포터
2. **진영(Faction) = 지역**: 같은 진영을 모으면 시너지 보너스
3. **등급(Rarity)**: 일반 → 희귀 → 영웅 → 전설

### 3.2 진영(지역)
| 지역 | 몬스터 예시 | 시너지 컨셉 |
|---|---|---|
| 🍄 버섯숲 | 슬라임·버섯·달팽이·나무정령 | 자연/재생 · 방치 자원↑ |
| 🧚 요정 마을 | 픽시·꼬마마녀·빛정령 | 마법 광역딜 |
| 🪨 바위 협곡 | 오크·골렘·돌거인 | 탱커/방어벽 |
| ⚙️ 태엽 도시 | 로봇·태엽곰·인형 | 기계 연계 버프 |
| ❄️ 눈꽃 설원 | 눈사람·펭귄·얼음정령 | 빙결/감속 |

- 전설 등급 = 각 지역 보스급 대형 몬스터(메이플 "머쉬맘" 감성)
- **저작권 주의**: 메이플의 이름/그림을 그대로 쓰지 않는다. "느낌"만 참고, 이름·에셋은 새로.

### 3.3 전투 (오토배틀)
- 스쿼드를 편성하고 시작하면 **자동 전투**로 진행 (조작 최소)
- 진영 시너지 + 역할 배치가 승패를 가름 → 조합의 재미
- 지휘관 액티브 스킬로 약간의 손맛 추가

### 3.4 방치(Idle) — 서브 시스템
- 오프라인/미접속 중에도 되찾은 지역에서 **자원 자동 수집**
- 복귀 시 누적 자원 수령 → 강화에 투입
- 펫이 방치 수집 효율을 높임 (3.7 참고)

### 3.5 지휘관(Commander) — 핵심
- 플레이어의 분신/정체성. 성장 대상
- **팀 전체 패시브 버프** (예: 특정 진영 공격력 +10%) → 스쿼드 구성에 전략성 부여
- **액티브 스킬 1개** — 자동 전투 중 발동
- 아레나에서 상대 식별자로 사용 ("○○ 지휘관의 부대")
- 비용 저렴(초상화 + 수치) → 가성비 좋아 MVP 포함

### 3.6 아레나 (비동기 PvP)
- 상대는 **실제 유저가 저장해둔 스쿼드 스냅샷** — 실시간 아님
- 상대 데이터를 읽어와 AI가 전투 진행 → **일반 DB로 충분, 실시간 서버 불필요**
- **초반 콜드스타트 해결**: 제작자가 디자인한 **봇 스쿼드 수십 개로 아레나 시딩** → 유저 증가 시 실제 스쿼드가 상대 풀에 자연 유입
- 랭킹·보상 제공

### 3.7 펫(Pet) — MVP는 미니멀
- **경제/편의 담당**: 오프라인 자원 자동 수집 +X%
- 전투는 안 하거나 소량만 → **전투 밸런스 미영향(안전)**
- **MVP 범위: 펫 1슬롯 + 방치 보너스만.** 진화/다수/애니는 반응 좋으면 확장

## 4. 기술 아키텍처

| 영역 | 선택 | 이유 |
|---|---|---|
| 제작 방식 | **웹 기술** (HTML/JS/TS) | 기존 실력 재사용, 새 엔진 학습 불필요 |
| 렌더링 | **Phaser** (2D 게임 라이브러리) | 스프라이트·전투 연출에 순수 DOM보다 적합, 여전히 JS/TS |
| 백엔드/저장 | **Firebase** (Auth + Firestore) | 덕뮤에서 쓰던 것 재사용, 비동기 아레나에 충분 |
| 배포 (웹) | 브라우저 우선 | 즉시 플레이, 설치 마찰 없음 |
| 배포 (모바일) | 추후 **Capacitor 래핑** | 같은 코드로 앱스토어 출시 가능 (선택) |
| 아트 | 유료/무료 픽셀 에셋 팩 (itch.io 등) | 직접 제작 회피 |

**핵심 원칙:** "웹 기술로 만든다" ≠ "브라우저에만 갇힌다". 웹으로 만들고 나중에 모바일로 래핑 가능.

## 5. 수익화 & 유통

### 5.1 수익화
- 1차: **광고** (게임 포털 rev-share + 리워드 광고 — 방치형은 부스트용 자발적 광고 시청 궁합 좋음)
- 2차(나중): 소액 결제 (뽑기 재화, 방치 부스트, 펫 등)

### 5.2 유통 (덕뮤가 실패한 지점 — 여기서 해결)
- **웹 게임 포털** (Poki, CrazyGames 등): 유통을 대신 해주고 광고수익 분배
- **incremental/방치 게임 커뮤니티** (r/incremental_games 등): 표적 유저가 새 웹 방치 게임을 스스로 찾아 플레이·공유 → 공짜 유통 채널

## 6. MVP 범위 (첫 출시)

### 6.1 포함 ✅
- 오토배틀 + 스쿼드 조합
- 방치 자원 파밍
- 지휘관 (패시브 버프 + 액티브 1개)
- 비동기 아레나 (봇 시딩)
- 펫 (1슬롯, 방치 보너스만)
- 1~2개 지역 + 각 지역 몬스터 5~6종 (전 지역을 다 만들지 않음)

### 6.2 제외 ❌ (나중)
- 펫 진화·다수·펫 전투
- 길드/소셜, 실시간 PvP
- 이벤트·시즌 운영
- 전 지역(5개) 전부 — 반응 보고 확장
- 결제 시스템 (광고 검증 후 도입)

## 7. 성공 기준
- **1차(창작)**: 제작자 본인이 "계속 하고 싶은" 첫 플레이 가능 빌드 완성
- **2차(검증)**: incremental 커뮤니티/포털에 올렸을 때 리텐션(재방문) 발생
- **3차(수익)**: 광고 수익 발생 시작 → 월 20만원 안팎 목표

| 리스크 | 대응 |

| 주 5시간 지속성 | 시스템(숫자) 중심 설계로 콘텐츠 제작 부담 최소화 |
